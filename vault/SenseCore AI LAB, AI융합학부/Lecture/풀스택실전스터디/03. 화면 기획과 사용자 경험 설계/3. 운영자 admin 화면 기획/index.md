# 3. 운영자 admin 화면 기획
앞 페이지에서 우리는 구매자 storefront의 흐름을 설계했습니다. 이제 운영자 화면을 따로 봐야 합니다. 많은 초급 프로젝트가 storefront만 잘 만들고 admin은 나중에 대충 붙입니다. 하지만 실제 서비스에서는 운영자 화면이 없으면 주문 이상 상태를 파악할 수 없고, 상품 현황도 볼 수 없으며, 결제 실패나 사용자 문의 대응도 느려집니다.

운영자 화면은 구매자 화면의 축소판이 아닙니다. 목적 자체가 다릅니다. 구매자는 "무엇을 사고 어떻게 주문을 완료할까"를 고민하고, 운영자는 "지금 서비스 상태가 어떤가, 어디에 문제가 있는가, 무엇을 먼저 봐야 하는가"를 고민합니다. 따라서 admin은 탐색보다 모니터링, 구매보다 판단, 매력적인 구성보다 정보 밀도와 우선순위가 중요합니다.

이번 페이지의 목적은 운영자가 빠르게 상태를 읽고 필요한 조치를 결정할 수 있도록, admin 영역의 핵심 화면과 정보 구조를 정리하는 것입니다.

# 왜 admin 화면을 따로 설계해야 하는가
운영자 화면을 별도 설계하지 않으면 보통 아래 두 가지 실수가 생깁니다.

- 구매자 화면 데이터를 그대로 재사용해 운영자 화면을 만든다
- 운영자가 실제로 무엇을 해야 하는지 정의되지 않은 상태에서 테이블만 만든다

이렇게 되면 admin은 금방 아래 문제를 드러냅니다.

- 주문 목록은 있는데 어떤 주문이 위험한지 보이지 않는다
- 결제 실패가 발생했는데 어디서 확인해야 하는지 애매하다
- 상품 현황은 보이지만 운영 관점 우선순위가 없다
- 고객 문의가 들어왔을 때 주문/상품/챗봇 흐름을 함께 볼 수 없다

따라서 admin은 반드시 "운영자가 무엇을 가장 빨리 알아야 하는가"를 중심으로 설계해야 합니다.

# 이 페이지에서 정할 것
이 페이지를 읽고 나면 아래 항목이 정리되어 있어야 합니다.

- 운영자 admin의 핵심 목표
- 필수 화면 목록
- 대시보드/주문/상품 현황의 역할 차이
- 결제 실패와 이상 상태를 보여 주는 방법
- 권한과 인증 경계

이 문서가 정리되면, 이후 admin 프론트 구현과 운영 API 설계가 자연스럽게 이어집니다.

# 1. 운영자 admin의 핵심 목표
이번 교재에서 운영자 화면은 아래 네 가지 질문에 답할 수 있어야 합니다.

- 지금 주문이 정상적으로 흐르고 있는가
- 실패하거나 막힌 주문은 없는가
- 상품 상태는 어떤가
- 사용자가 문의했을 때 어떤 맥락을 함께 봐야 하는가

이 네 질문이 admin 화면 구조의 기준입니다.

# 2. 이번 교재의 admin 핵심 화면
이번 MVP에서 반드시 설계해야 하는 admin 화면은 아래와 같습니다.

- 운영자 로그인 화면
- 운영 대시보드
- 주문 목록 화면
- 주문 상세 또는 상태 확인 화면
- 상품 현황 화면

이번 교재에서는 복잡한 운영 기능보다 "핵심 운영 상태를 읽는 최소 surface"를 만드는 데 집중합니다. 따라서 아래 화면은 이번 범위에서 제외해도 됩니다.

- 세밀한 권한 관리
- 프로모션 관리
- 배너/콘텐츠 관리
- 사용자 계정 상세 관리
- 정산/회계 화면

# 3. 운영자 로그인 화면
운영자 로그인은 단순히 인증 폼이 아니라 admin 영역과 storefront 영역을 분리하는 첫 경계입니다.

## 목적
- 운영자만 admin에 들어오게 한다
- storefront 사용자와 운영자 공간을 분리한다

## 꼭 필요한 요소
- 운영자 인증 입력
- 로그인 성공 후 `/admin`으로 이동
- 실패 시 명확한 피드백

## 설계 원칙
- storefront 로그인과 혼동되지 않아야 한다
- 운영자용 레이아웃 시작점이라는 인식이 있어야 한다

# 4. 운영 대시보드
대시보드는 admin의 첫 화면이며, 운영자가 가장 먼저 봐야 할 요약 정보를 담습니다.

## 목적
- 서비스 상태를 한눈에 파악한다
- 어떤 화면으로 더 들어가야 하는지 우선순위를 제공한다

## 꼭 필요한 요소
- 전체 주문 수 또는 최근 주문 수
- 상태별 주문 요약
- 실패 결제 또는 이상 상태 요약
- 최근 주문 진입
- 상품 현황 요약

## 대시보드의 핵심 원칙
- 모든 데이터를 다 보여 주는 것이 목적이 아니다
- 지금 무엇을 먼저 봐야 하는지 판단하게 하는 것이 목적이다

즉 대시보드는 상세 테이블의 축약판이 아니라, 운영 우선순위 화면입니다.

# 5. 주문 목록 화면
주문 목록은 운영자가 가장 자주 보는 화면이 됩니다.

## 목적
- 주문 상태를 빠르게 훑는다
- 문제 주문을 찾아낸다
- 주문 상세로 들어간다

## 꼭 필요한 요소
- 주문 번호
- 사용자 또는 구매자 식별 정보
- 주문 상태
- 결제 상태
- 생성 시각
- 상세 진입

## 설계 원칙
- 운영자는 상품 카드보다 상태 기반 테이블을 더 빠르게 읽는다
- 상태 정보는 눈에 잘 띄어야 한다

## 중요하게 보여야 하는 것
- `PAYMENT_FAILED`
- `PENDING`
- 비정상 흐름처럼 보이는 상태

운영자 화면에서는 이런 상태가 일반 성공 상태보다 더 우선적으로 눈에 들어와야 합니다.

# 6. 주문 상세 또는 상태 확인 화면
주문 상세 화면은 운영자가 한 건의 주문을 정확히 이해하는 화면입니다.

## 목적
- 주문 상태를 구체적으로 확인한다
- 결제 결과를 함께 본다
- 상품 구성과 사용자 맥락을 확인한다

## 꼭 필요한 요소
- 주문 기본 정보
- 주문 항목 목록
- 총액
- 주문 상태
- 결제 상태
- 생성 시각/변경 시각

## 설계 원칙
- 구매자용 주문 상세보다 운영자용 상세는 상태 해석이 더 중요하다
- "이 주문이 왜 지금 이 상태인가"를 이해할 수 있어야 한다

# 7. 상품 현황 화면
이번 교재에서 상품 관리는 CRUD 전체를 다루기보다, 운영자가 현재 상품 상태를 읽는 수준으로 제한합니다.

## 목적
- 어떤 상품이 노출/판매 중인지 본다
- 운영자가 상품 현황을 빠르게 확인한다

## 꼭 필요한 요소
- 상품명
- 가격
- 상태
- 간단한 목록/정렬

## 설계 원칙
- 상품 운영 전체 관리보다 현황 파악이 우선이다
- 이번 MVP에서는 주문 모니터링보다 우선순위가 낮다

# 8. admin에서 결제 실패와 이상 상태를 어떻게 보여 줄 것인가
운영자 화면의 품질은 정상 상태를 얼마나 잘 보여 주느냐보다, 실패 상태를 얼마나 빨리 식별하게 하느냐에 더 크게 좌우됩니다.

## 반드시 드러나야 하는 상태
- 결제 실패
- 주문 생성 후 비정상적으로 멈춘 상태
- 확인이 필요한 예외 상태

## 설계 방식
- 대시보드 요약 카드
- 주문 목록 상태 배지
- 주문 상세의 상태 히스토리 또는 상태 구역

즉 실패 상태는 테이블 안에 묻히면 안 되고, 요약과 상세 양쪽에서 모두 보이게 해야 합니다.

# 9. 고객 문의와 운영 흐름의 연결
이번 교재에는 AI 상담 챗봇이 포함되므로, 운영자는 주문과 상품 상태만 보지 않고 문의 맥락도 이해할 필요가 있습니다.

이번 MVP에서 admin에 꼭 별도 문의센터를 만들 필요는 없지만, 최소한 아래 사고는 가져가야 합니다.

- 상품 관련 문의는 상품 상세와 연결된다
- 주문 관련 문의는 주문 상태와 연결된다
- 챗봇이 해결하지 못한 문의는 운영 판단이 필요할 수 있다

즉 admin은 독립된 표 관리 화면이 아니라, 실제 사용자 문제를 이해하는 운영 surface입니다.

# 10. 권한과 역할
이번 교재에서는 세밀한 역할 분리는 하지 않지만, 최소한 아래 원칙은 필요합니다.

## 권한 원칙
- 일반 구매자는 `/admin`에 접근할 수 없다
- 운영자 인증이 없는 상태에서 admin 라우트는 보호된다

## 역할 설계는 왜 최소화하는가
이번 MVP에서는 권한 체계를 세세하게 설계하기보다, 운영자 전용 공간을 분리하는 데 집중하는 편이 좋습니다. 복잡한 role matrix는 이후 확장 주제로 남겨도 충분합니다.

# 11. 실습용 admin 화면 목록
이번 교재를 따라 하는 학습자는 최소한 아래 화면 목록을 먼저 적어 두면 충분합니다.

| 화면 | 목적 |
| --- | --- |
| 운영자 로그인 | admin 진입 |
| 대시보드 | 핵심 상태 요약 |
| 주문 목록 | 상태 모니터링 |
| 주문 상세 | 한 건 상태 해석 |
| 상품 현황 | 상품 목록 확인 |

# 12. storefront와 다른 점을 분명히 해야 한다
storefront는 구매 결정을 돕는 화면이고, admin은 운영 판단을 돕는 화면입니다. 따라서 아래 차이가 명확해야 합니다.

| 구분 | storefront | admin |
| --- | --- | --- |
| 목적 | 구매 전환 | 상태 모니터링 |
| 주요 UI | 카드/상세/CTA | 테이블/요약/상태 배지 |
| 핵심 질문 | 무엇을 살까 | 어디가 문제인가 |
| 우선 정보 | 상품 이해 | 상태와 예외 |

이 차이가 보이지 않으면 admin이 storefront의 표 버전처럼 되어 버립니다.

# 13. 자주 하는 실수
- 운영자 대시보드를 단순 통계 카드 모음으로만 만든다
- 주문 목록에서 실패 상태가 눈에 띄지 않게 만든다
- 주문 상세를 구매자용 상세와 거의 비슷하게 만든다
- 상품 현황 화면에 CRUD 전체를 한 번에 넣으려 한다
- 운영자 화면과 구매자 화면의 목적 차이를 무시한다

이 다섯 가지 실수는 admin을 "있기만 한 화면"으로 만듭니다.

# 정리
이 페이지에서 우리는 운영자 admin의 핵심 목표와 필수 화면을 정리했습니다. 대시보드, 주문 목록, 주문 상세, 상품 현황이 각각 어떤 역할을 갖는지 분명해졌고, 실패 상태와 이상 상태를 어떻게 드러낼지도 정리했습니다. 다음 페이지에서는 마지막으로 AI 상담 챗봇의 대화 흐름과 화면 위치를 설계해, 사용자 지원 흐름까지 완성합니다.
