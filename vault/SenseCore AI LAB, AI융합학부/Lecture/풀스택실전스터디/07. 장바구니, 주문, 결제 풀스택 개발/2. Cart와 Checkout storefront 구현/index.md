# 2. Cart와 Checkout storefront 구현
앞 페이지에서 `Cart` 도메인과 금액 계산 규칙을 백엔드에 세웠다면, 이제 그 결과를 사용자가 실제로 조작하는 화면에 붙여야 합니다. 이 단계에서 `storefront`는 단순 탐색 앱에서 벗어나, 사용자가 "구매를 준비하는 상태"를 직접 조작하는 앱으로 넘어갑니다.

이번 페이지의 핵심은 장바구니 목록을 한 번 보여주는 데서 끝나지 않습니다. 상품 담기, 수량 변경, 항목 제거, 합계 갱신, checkout 진입, 주문 직전 확인까지 하나의 흐름으로 연결해야 합니다. 그래야 다음 페이지의 주문 생성과 결제 확인이 자연스럽게 이어집니다.

즉 이번 장은 `Cart` 백엔드를 `storefront`의 첫 본격적인 상태 변경 UX로 바꾸는 단계입니다.

# 왜 Cart 화면과 Checkout을 같이 다루는가
장바구니와 checkout은 서로 다른 화면이지만, 사용자 입장에서는 거의 하나의 흐름입니다.

- 장바구니에 담는다
- 수량을 조정한다
- 총액을 확인한다
- checkout으로 이동한다
- 주문 직전 내용을 다시 확인한다

장바구니만 있고 checkout이 없으면 다음 행동이 비고, checkout만 있으면 어떤 상태를 확인하는지 맥락이 없습니다. 따라서 이번 교재에서는 두 화면을 함께 설명하는 것이 자연스럽습니다.

# 이 페이지에서 정할 것
이 페이지를 읽고 나면 아래 항목이 정리되어 있어야 합니다.

- 장바구니 화면은 어떤 정보를 보여줘야 하는가
- 사용자는 어떤 방식으로 수량과 항목을 수정하는가
- 합계는 어떻게 반영되고 언제 다시 계산되는가
- checkout 화면은 어떤 확인 역할을 해야 하는가
- 주문 생성 직전 어떤 검증과 안내가 필요한가

# 1. 이번 단계에서 storefront가 반드시 할 수 있어야 하는 것
이번 단계의 `storefront`는 최소한 아래 상태까지 와 있어야 합니다.

- 상품을 장바구니에 담는다
- 담긴 상품을 본다
- 수량을 바꾼다
- 항목을 제거한다
- 총액 변화를 확인한다
- checkout 화면으로 이동한다

즉 장바구니가 단순 데이터 뷰가 아니라, 사용자가 실제로 조작하는 구매 준비 화면이어야 합니다.

# 2. Cart 화면의 첫 책임은 무엇인가
장바구니 화면은 "무엇이 담겨 있는가"와 "지금 수정할 수 있는가"를 동시에 보여줘야 합니다. 따라서 초기 장바구니 화면은 아래 정도에 집중하는 것이 좋습니다.

- 담긴 상품 목록
- 각 상품의 이름과 가격
- 현재 수량
- 항목 금액
- 총액
- 항목 삭제 또는 수량 변경 동작

중요한 것은 장바구니 화면이 단순 요약이 아니라, 수정 가능한 작업 화면이라는 점입니다.

# 3. 수량 변경 UX는 왜 특히 중요할까
장바구니에서 가장 자주 일어나는 상호작용은 수량 변경입니다. 이 UX가 거칠면 바로 사용자 경험이 무너집니다.

이번 교재에서는 최소한 아래를 먼저 정하는 것이 좋습니다.

- 증가 버튼 또는 입력 방식
- 감소 버튼 또는 직접 수정 방식
- 1 미만으로 내려갈 때의 처리
- 변경 직후 합계 반영 방식

이 흐름이 자연스러워야 사용자는 장바구니를 "수정 가능한 상태"로 인식합니다.

# 4. 항목 제거는 어떻게 보여줘야 하는가
항목 제거도 단순 삭제 버튼 하나로 끝나지 않습니다. 아래 질문에 답할 수 있어야 합니다.

- 항목 제거 후 화면은 즉시 갱신되는가
- 합계는 같이 갱신되는가
- 마지막 항목을 지우면 empty 상태로 전환되는가

이 세 가지가 맞지 않으면 장바구니는 금방 불안정하게 보입니다.

# 5. Cart 화면에서 empty 상태는 왜 더 중요해지는가
탐색 화면의 empty 상태와 장바구니의 empty 상태는 의미가 다릅니다. 장바구니가 비어 있다는 것은 "다음 행동으로 갈 수 없다"는 뜻이기도 하기 때문입니다.

이번 단계에서는 empty 상태가 아래 역할을 해야 합니다.

- 장바구니가 비어 있음을 명확히 알려준다
- 다시 상품 탐색으로 이동할 수 있게 한다
- checkout을 막는다

즉 empty 상태는 단순한 안내 문구가 아니라, 구매 흐름의 분기점입니다.

# 6. 합계는 언제 갱신되어야 하는가
장바구니 UX에서 합계는 사용자의 신뢰와 직결됩니다. 따라서 합계는 아래 순간마다 일관되게 갱신되어야 합니다.

- 상품 추가 후
- 수량 변경 후
- 항목 제거 후
- 장바구니 재조회 후

중요한 것은 프론트가 합계를 추측하지 않고, 서버 기준 계산과 같은 결과를 보여줘야 한다는 점입니다.

# 7. 화면에서 계산을 얼마나 해도 될까
프론트는 장바구니 금액을 표현할 수는 있지만, 최종 기준은 서버가 가져야 합니다. 이번 단계에서는 아래처럼 구분하는 것이 좋습니다.

## 프론트에서 해도 되는 것
- 금액 포맷팅
- 수량 변경 직후의 낙관적 UI 보조

## 서버 기준으로 유지해야 하는 것
- 최종 항목 금액
- 총액
- 유효하지 않은 수량 차단 결과

즉 화면은 빠른 피드백을 줄 수 있지만, 최종 계산 진실은 서버에 있어야 합니다.

# 8. Checkout 화면의 첫 책임은 무엇인가
checkout 화면은 아직 결제를 실행하는 장이 아니더라도, 아래 역할을 먼저 해야 합니다.

- 장바구니 내용을 다시 요약한다
- 주문 직전 확인 정보를 보여준다
- 다음 단계인 주문 생성으로 넘어갈 준비를 한다

즉 checkout은 cart의 복사본이 아니라, "이제 이 내용으로 주문을 만들겠다"는 확인 단계입니다.

# 9. Cart와 Checkout은 어떤 정보 차이가 있어야 하는가
두 화면이 완전히 같아 보이면 역할이 분리되지 않습니다.

## Cart 화면이 더 강한 것
- 수량 수정
- 항목 제거
- 계속 쇼핑 이동

## Checkout 화면이 더 강한 것
- 최종 요약
- 주문 직전 확인 문맥
- 다음 단계 CTA

이 차이가 있어야 사용자는 두 화면을 각각 다른 목적의 단계로 이해합니다.

# 10. checkout 진입은 언제 막아야 하는가
checkout은 언제나 열리게 두면 안 됩니다. 최소한 아래 상황에서는 막거나 안내가 필요합니다.

- 비어 있는 장바구니
- 인증되지 않은 사용자
- 유효하지 않은 수량 상태

이 기준이 있어야 다음 페이지의 주문 생성 API가 불필요한 실패를 덜 겪습니다.

# 11. 로그인과 장바구니 흐름은 어떻게 연결되는가
이번 교재에서는 cart API가 인증 기반이므로, 장바구니와 checkout은 로그인 상태와 직접 연결됩니다. 따라서 아래를 먼저 생각해야 합니다.

- 로그인하지 않은 상태에서 장바구니 진입 시 어떻게 할 것인가
- 로그인 후 다시 원래 cart/checkout 경로로 돌아올 것인가
- 세션 만료 시 cart 화면은 어떻게 반응할 것인가

이 흐름이 자연스러워야 장바구니가 실제 구매 준비 단계로 기능합니다.

# 12. 장바구니 화면의 오류는 어떻게 보여줘야 하는가
이번 단계에서 최소한 아래 오류는 구분되어 보여야 합니다.

- cart 조회 실패
- 수량 변경 실패
- 항목 제거 실패
- 인증 실패

조회 실패는 화면 전체 상태일 수 있고, 수량 변경 실패는 항목 단위 메시지일 수 있습니다. 즉 모든 오류를 같은 박스로 뭉개면 UX가 거칠어집니다.

# 13. 주문 직전 검증은 무엇을 먼저 보여줘야 하는가
아직 주문 생성은 다음 장이지만, checkout 단계에서 최소한 아래는 확인할 수 있어야 합니다.

- 무엇을 주문하려는가
- 몇 개를 주문하려는가
- 총액이 얼마인가
- 로그인된 사용자인가

이 정보가 있어야 다음 페이지에서 주문 생성 버튼이 의미를 갖습니다.

# 14. 이번 장의 구현 완료 기준은 무엇인가
이번 장에서 완료라고 볼 수 있는 상태는 아래 정도입니다.

- 장바구니 화면이 실제 API를 통해 렌더링된다
- 수량 변경과 항목 제거가 된다
- 합계가 서버 기준으로 갱신된다
- empty/error/auth 상태가 있다
- checkout 화면이 cart 요약을 기반으로 열린다

즉 실제 결제는 없어도, 구매 준비 화면은 성립해야 합니다.

# 15. 이번 페이지를 실제 구현 산출물로 바꾸면 무엇이 나와야 하나
이번 페이지를 코드와 실행 결과로 옮기면 최소한 아래 산출물이 있어야 합니다.

- `storefront` cart 페이지
- `storefront` checkout 요약 페이지
- cart 조회/수정/삭제 API 연결
- 합계 렌더링 UI
- empty/error/auth 분기 UI
- checkout 진입 흐름

이 산출물이 있어야 다음 페이지에서 order 생성과 payment 확인을 실제 백엔드 워크플로로 연결할 수 있습니다.

# 16. 자주 하는 실수
- 장바구니를 읽기 전용 목록처럼 만들고 수정 UX를 약하게 둔다
- 수량 변경 후 합계 갱신을 늦추거나 빼먹는다
- empty 상태에서도 checkout 버튼을 열어 둔다
- cart와 checkout 화면이 거의 동일해 역할 차이가 없다
- 로그인 만료와 cart 조회 실패를 같은 오류로 처리한다
- 프론트에서만 합계를 계산하고 서버 기준을 무시한다

이 여섯 가지 실수는 다음 장의 주문 생성과 결제 흐름을 바로 불안정하게 만듭니다.

# 정리
이 페이지에서 우리는 `Cart` 백엔드를 `storefront` 장바구니 화면과 checkout 요약 UI에 연결해, 사용자가 실제로 구매 준비 상태를 조작하는 흐름을 만드는 방법을 정리했습니다. 또한 장바구니와 checkout은 하나의 흐름이지만 서로 다른 역할을 가진 화면이며, 합계 갱신과 인증 상태와 주문 직전 검증이 함께 맞물려야 한다는 점도 확인했습니다.
