# 2. 사용자, 액터, 핵심 시나리오
1강에서 우리는 왜 이 서비스를 만들고, 어디까지를 성공으로 볼지 정했습니다. 이제 그 목표를 실제 동작 흐름으로 바꿔야 합니다. 서비스는 추상적인 비전으로 구현되지 않습니다. 결국 누가 들어와서 어떤 행동을 하고, 그 과정에서 어떤 상태가 바뀌는지로 구현됩니다.

이 페이지의 목적은 단순히 사용자 종류를 나열하는 것이 아닙니다. 누가 시스템을 사용하는지, 어떤 외부 주체가 시스템과 상호작용하는지, 어떤 흐름을 반드시 완성해야 하는지를 고정해 뒤의 DDD 모델링과 화면 설계의 입력값으로 만드는 것입니다.

# 왜 이 페이지가 중요한가
개발 강의에서 자주 생기는 문제는 기능 목록은 많은데 정작 "누가 무엇을 끝내야 하는가"가 불분명한 상태로 구현을 시작하는 것입니다. 이 경우 화면은 늘어나지만 핵심 흐름은 완성되지 않습니다.

예를 들어 아래와 같은 상황이 발생합니다.

- 상품 목록 화면은 있는데 비회원과 회원의 차이가 정리되어 있지 않다
- 운영자 페이지는 있는데 운영자가 실제로 무엇을 봐야 하는지 정의되지 않았다
- AI 챗봇은 붙였지만 어떤 질문에 답하고 어디서 사람 상담으로 넘길지 정하지 않았다
- 결제 API는 만들었지만 누가 어떤 시점에 결제 성공과 실패를 확인해야 하는지 설명할 수 없다

그래서 이 페이지에서는 먼저 세 가지를 구분합니다.

1. 사용자: 제품의 가치를 직접 얻는 사람
2. 액터: 시스템과 상호작용하는 모든 주체
3. 핵심 시나리오: 반드시 끝까지 완성해야 하는 행동 흐름

이 셋을 구분해야 뒤에서 바운디드 컨텍스트, 화면, API, 테스트 시나리오가 자연스럽게 이어집니다.

# 이 페이지에서 정할 것
이 페이지를 읽고 나면 아래 항목이 문서로 정리되어 있어야 합니다.

- 주요 사용자 집단
- 시스템과 상호작용하는 액터 목록
- 각 액터의 목표와 책임
- 구매 흐름, 운영 흐름, 상담 흐름의 핵심 시나리오
- 이번 강의에서 반드시 완성할 대표 시나리오

이 문서가 없으면 다음 장의 DDD 모델링은 추상적인 구조 설명으로 끝나기 쉽습니다.

# 1. 사용자와 액터를 구분하기
먼저 사용자와 액터를 구분해야 합니다. 두 용어를 섞으면 도메인 경계가 흐려집니다.

## 사용자
사용자는 이 서비스의 가치를 직접 얻는 사람입니다. 이 강의에서 주요 사용자는 아래 두 집단입니다.

- 구매자
- 운영자

## 액터
액터는 시스템과 상호작용하는 모든 주체입니다. 사람일 수도 있고 외부 시스템일 수도 있습니다. 따라서 이 강의의 액터는 사용자보다 범위가 넓습니다.

- 비회원 방문자
- 회원 구매자
- 운영자
- 외부 결제 시스템
- AI 상담 챗봇
- 알림 또는 메시지 발송 시스템

이 차이를 이해해야 합니다. 예를 들어 결제 시스템은 사용자가 아니지만 중요한 액터입니다. 챗봇도 최종 사용자는 아니지만 별도 의사결정과 응답을 수행하므로 액터로 다뤄야 합니다.

# 2. 이번 강의의 주요 사용자 정의
이 강의에서는 실제 이커머스 서비스를 단순화하되, 역할 차이는 분명하게 유지합니다.

## 2.1 비회원 방문자
비회원 방문자는 아직 계정을 만들지 않았지만 상품을 탐색하고 서비스 가치를 판단하는 주체입니다.

### 주요 목표
- 상품 목록을 탐색한다
- 특정 상품의 상세 정보를 확인한다
- 회원가입 여부를 결정한다

### 시스템에 기대하는 것
- 상품 탐색이 빠를 것
- 상품 정보가 이해 가능할 것
- 로그인 전에도 기본 탐색이 가능할 것

### 제한 사항
- 장바구니 저장이나 주문 생성은 제한될 수 있다
- 개인 주문 이력은 볼 수 없다

비회원 흐름을 별도로 잡는 이유는 카탈로그와 인증 경계를 나중에 분리해서 생각하기 위함입니다.

## 2.2 회원 구매자
회원 구매자는 이 강의의 핵심 사용자입니다. 우리가 반드시 완성해야 하는 대표 시나리오는 대부분 이 사용자 기준으로 정의됩니다.

### 주요 목표
- 상품을 비교하고 선택한다
- 장바구니에 담는다
- 주문을 생성하고 결제를 진행한다
- 주문 결과와 상태를 확인한다
- 필요 시 챗봇으로 문의를 시작한다

### 시스템에 기대하는 것
- 로그인 이후 상태가 안정적으로 유지될 것
- 장바구니와 주문 합계가 정확할 것
- 결제 결과가 명확하게 보일 것
- 주문 이후 상태가 추적 가능할 것

### 주의할 점
구매자는 단순히 "프론트 사용자"가 아닙니다. 상품 탐색, 장바구니, 주문, 결제, 상담의 중심 액터입니다. 즉 구매자 기준 흐름이 흔들리면 전체 서비스가 흔들립니다.

## 2.3 운영자
운영자는 구매자와 다른 목표를 가집니다. 운영자는 물건을 사기 위해 들어오지 않고, 서비스 상태를 관리하기 위해 들어옵니다.

### 주요 목표
- 상품 현황을 확인한다
- 주문 상태를 확인하고 관리한다
- 결제 이상 상태를 파악한다
- 고객 문의 흐름을 본다

### 시스템에 기대하는 것
- 핵심 운영 지표가 한 화면에 보여야 한다
- 주문 상태를 빠르게 필터링할 수 있어야 한다
- 실패 케이스를 놓치지 않아야 한다

### 왜 별도 사용자로 다뤄야 하는가
운영자 화면은 구매자 화면의 연장이 아닙니다. 따라서 뒤에서 admin surface를 별도 장으로 나눌 근거가 여기에서 만들어집니다.

# 3. 주요 액터 정의
이제 사용자 외의 액터를 정리합니다. 이 단계는 나중에 도메인 이벤트와 외부 연동 경계를 잡는 데 중요합니다.

## 3.1 외부 결제 시스템
이번 강의에서는 실제 PG 대신 모의 결제 흐름을 사용하더라도, 개념적으로는 외부 결제 시스템을 별도 액터로 다룹니다.

### 역할
- 결제 승인 또는 실패 결과를 반환한다
- 주문 생성 이후 결제 상태 변화를 유발한다

### 설계에 주는 영향
- Order와 Payment를 분리해서 생각해야 한다
- 결제 성공과 주문 확정을 같은 개념으로 취급하면 안 된다

## 3.2 AI 상담 챗봇
챗봇은 단순한 UI 위젯이 아니라 별도의 액터입니다. 고객 문의를 받아 응답하고, 필요한 경우 사람 상담 또는 운영 흐름으로 넘기는 역할을 수행합니다.

### 역할
- 상품 관련 질문에 답한다
- 주문 상태 관련 문의를 받는다
- FAQ를 안내한다
- 해결이 어려운 경우 상담 전환을 유도한다

### 설계에 주는 영향
- 상품 정보, 주문 정보, FAQ 정보에 접근하는 읽기 경로가 필요하다
- 무제한 자동응답이 아니라 안전한 핸드오프 경계가 필요하다

## 3.3 알림 시스템
이번 강의에서 알림을 풀 구현하지 않더라도, 주문 생성과 결제 상태 변화에는 알림 액터가 개념적으로 존재합니다.

### 역할
- 주문 생성 또는 결제 결과를 사용자에게 알린다
- 운영자에게 이상 상태를 전달할 수 있다

### 설계에 주는 영향
- 상태 변화가 단순 화면 반영으로 끝나는지, 후속 액션을 유발하는지 생각하게 만든다

# 4. 핵심 시나리오를 정의하는 방법
핵심 시나리오는 "사용자가 하고 싶은 것"을 시스템 흐름으로 풀어 쓴 문서입니다. 좋은 시나리오는 아래 질문에 답할 수 있어야 합니다.

- 누가 시작하는가
- 어떤 전제조건이 필요한가
- 어떤 행동을 순서대로 하는가
- 시스템은 각 단계에서 무엇을 보여주거나 처리하는가
- 완료 조건은 무엇인가
- 실패 시 어떤 분기가 생기는가

이 강의에서는 기능 목록보다 시나리오를 먼저 고정합니다. 이유는 간단합니다. 기능은 시나리오를 완성하기 위해 존재하기 때문입니다.

# 5. 반드시 완성해야 하는 대표 시나리오
이번 강의에서 최소한 아래 다섯 개 시나리오는 완성되어야 합니다.

## 5.1 구매자 상품 탐색 시나리오
### 액터
- 비회원 방문자 또는 회원 구매자

### 시작점
- 사용자가 storefront에 진입한다

### 흐름
1. 사용자는 상품 목록을 본다.
2. 카테고리 또는 상품 카드를 통해 관심 상품을 찾는다.
3. 상품 상세 페이지로 이동한다.
4. 가격, 설명, 옵션, 재고 상태를 확인한다.

### 완료 조건
- 사용자가 특정 상품을 구매 후보로 선택할 수 있다

### 이 시나리오가 중요한 이유
이 흐름이 없으면 장바구니와 주문은 존재 의미가 없습니다. 따라서 Catalog 컨텍스트와 storefront 정보 구조는 이 시나리오를 중심으로 설계해야 합니다.

## 5.2 회원 구매자 장바구니 시나리오
### 액터
- 회원 구매자

### 전제조건
- 사용자가 로그인 상태다
- 상품 상세에서 구매 의사가 생겼다

### 흐름
1. 사용자는 상품을 장바구니에 담는다.
2. 장바구니 페이지에서 수량을 변경한다.
3. 시스템은 합계를 다시 계산한다.
4. 사용자는 주문 진행 버튼을 누른다.

### 완료 조건
- 주문 생성 직전의 구매 상태가 안정적으로 준비된다

### 실패 분기
- 인증이 없으면 로그인으로 유도한다
- 재고 또는 수량 조건이 맞지 않으면 수정 피드백을 준다

### 설계상 의미
이 시나리오가 있어야 Cart를 별도 컨텍스트 또는 명확한 모듈로 다룰 근거가 생깁니다.

## 5.3 주문 생성과 결제 시나리오
### 액터
- 회원 구매자
- 외부 결제 시스템

### 전제조건
- 장바구니에 유효한 상품이 있다

### 흐름
1. 사용자는 주문서를 확인한다.
2. 배송 정보와 결제 수단을 입력하거나 선택한다.
3. 시스템은 주문을 생성한다.
4. 결제 요청이 외부 결제 흐름으로 전달된다.
5. 결제 성공 또는 실패 결과가 돌아온다.
6. 시스템은 주문 상태를 갱신한다.

### 완료 조건
- 성공 시 주문 완료 상태가 보인다
- 실패 시 재시도 또는 복구 경로가 보인다

### 설계상 의미
이 시나리오는 이커머스의 핵심입니다. 이 흐름이 깨지면 서비스는 완성되지 않은 것입니다. 뒤에서 Order와 Payment를 나누는 이유도 바로 이 시나리오 때문입니다.

## 5.4 운영자 주문 모니터링 시나리오
### 액터
- 운영자

### 시작점
- 운영자가 admin에 로그인한다

### 흐름
1. 운영자는 대시보드에서 주문 현황을 본다.
2. 상태별 주문 수, 최근 주문, 실패 결제 건을 확인한다.
3. 특정 주문을 찾아 상세 상태를 확인한다.

### 완료 조건
- 운영자가 서비스 상태를 빠르게 파악할 수 있다

### 설계상 의미
운영자 기능을 단순 부가 기능이 아니라 별도 사용자 여정으로 다뤄야 한다는 근거가 됩니다.

## 5.5 AI 상담 챗봇 문의 시나리오
### 액터
- 회원 구매자 또는 비회원 방문자
- AI 상담 챗봇

### 흐름
1. 사용자가 챗봇 패널을 연다.
2. 상품 또는 주문 관련 질문을 입력한다.
3. 챗봇은 FAQ, 상품 정보, 주문 상태 관련 응답을 시도한다.
4. 해결이 어려우면 상담 전환 또는 운영 문의 경로를 제안한다.

### 완료 조건
- 사용자가 질문에 대한 1차 응답을 받는다
- 자동응답이 부족할 경우 다음 행동이 제시된다

### 설계상 의미
챗봇은 "있으면 좋은 기능"이 아니라 운영 효율과 사용자 지원 흐름의 일부입니다. 따라서 나중 장에서 AI 챗봇을 별도 부록이 아닌 본 과정으로 다루게 됩니다.

# 6. 사용자 여정을 한 장으로 요약하기
이 시점에서는 상세 플로우차트보다 요약된 사용자 여정이 더 중요합니다. 아래처럼 정리할 수 있어야 합니다.

| 사용자/액터 | 진입 이유 | 핵심 행동 | 완료 상태 |
| --- | --- | --- | --- |
| 비회원 방문자 | 상품을 둘러본다 | 목록 조회, 상세 확인 | 회원가입 또는 이탈 판단 |
| 회원 구매자 | 상품을 구매한다 | 로그인, 장바구니, 주문, 결제 | 주문 완료 또는 실패 확인 |
| 운영자 | 상태를 관리한다 | 대시보드 확인, 주문 모니터링 | 이상 상태 파악 |
| AI 상담 챗봇 | 문의를 처리한다 | 질문 수신, 응답, 전환 제안 | FAQ 해결 또는 핸드오프 |
| 결제 시스템 | 결제 결과를 반환한다 | 승인/실패 응답 | 주문 상태 변화 유발 |

이 표가 중요한 이유는, 뒤에서 화면 목록과 API 목록을 만들 때 이 표를 기준으로 빠진 흐름을 찾을 수 있기 때문입니다.

# 7. 이 페이지의 결과가 DDD로 어떻게 이어지는가
이 페이지는 곧바로 다음 장의 모델링 재료가 됩니다.

## 사용자와 액터는 바운디드 컨텍스트 경계의 힌트다
- 구매자 중심 흐름 -> Catalog, Cart, Order
- 운영자 중심 흐름 -> Admin/Operations
- 결제 시스템 -> Payment
- 챗봇 흐름 -> Support Assistant

## 핵심 시나리오는 애그리거트의 책임을 드러낸다
- 장바구니 수량 변경 -> Cart의 책임
- 주문 생성 -> Order의 책임
- 결제 상태 반영 -> Payment와 Order의 협력
- 상담 전환 -> Support Assistant와 Operations의 연결

## 실패 분기는 도메인 규칙의 단서다
- 인증이 없으면 주문으로 갈 수 없다
- 결제 실패는 주문 완료와 다르다
- 챗봇이 해결하지 못하면 사람 흐름으로 넘겨야 한다

즉 시나리오는 스토리텔링이 아니라 도메인 경계 발견 도구입니다.

# 8. 화면 기획과 API 설계로의 연결
시나리오가 정리되면 다음 단계에서 화면과 API를 자연스럽게 뽑아낼 수 있습니다.

예를 들어:

- 상품 탐색 시나리오 -> 상품 목록 화면, 상품 상세 화면, 상품 조회 API
- 장바구니 시나리오 -> 장바구니 화면, 담기/수정 API, 합계 계산 규칙
- 주문/결제 시나리오 -> 주문서 화면, 주문 생성 API, 결제 확인 API
- 운영자 시나리오 -> 관리자 대시보드, 주문 목록 API
- 챗봇 시나리오 -> 채팅 패널, FAQ 검색 또는 응답 API, 전환 이벤트

이 연결이 보이지 않는다면 아직 시나리오가 충분히 구체적이지 않은 것입니다.
