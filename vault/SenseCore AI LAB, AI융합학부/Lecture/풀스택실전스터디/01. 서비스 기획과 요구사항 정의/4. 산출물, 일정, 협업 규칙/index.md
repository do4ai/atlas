# 4. 산출물, 일정, 협업 규칙
1강부터 3강까지에서 우리는 왜 이 서비스를 만드는지, 누가 어떤 시나리오를 수행하는지, 이번 교재의 MVP가 어디까지인지를 정했습니다. 이제 남은 것은 실행 기준을 고정하는 일입니다. 실제 프로젝트는 좋은 아이디어만으로 굴러가지 않습니다. 어떤 문서가 먼저 필요하고, 어떤 코드가 뒤따라야 하며, 어느 시점에 무엇을 검증해야 하는지가 정리되어 있어야 합니다.

이 페이지의 목적은 팀이 같은 그림을 보게 만드는 것입니다. 기획 문서, 도메인 모델, 화면 설계, 구현, 검증, 배포까지 어떤 순서로 남아야 하는지 고정해 두면, 뒤 장에서 프론트와 백엔드가 따로 놀지 않고 한 줄로 연결됩니다.

# 왜 이 페이지가 필요한가
개발 강의에서 흔히 생기는 문제는 방향은 맞는데 실행 규칙이 없어서 중간부터 흐트러지는 것입니다.

예를 들어:

- 누군가는 화면부터 만들고, 누군가는 API부터 만든다
- 요구사항이 바뀌었는데 문서와 코드 중 무엇을 먼저 고쳐야 하는지 모른다
- 테스트는 나중에 하자고 미루다가 결국 핵심 흐름이 검증되지 않는다
- 배포는 했지만 무엇을 확인해야 하는지 기준이 없다

이런 문제를 막으려면 "무엇을 만들 것인가"뿐 아니라 "어떤 산출물을 어떤 순서로 만들 것인가"를 정해야 합니다. 그래서 이번 페이지는 일종의 프로젝트 운영 계약서 역할을 합니다.

# 이 페이지에서 정할 것
이 페이지를 읽고 나면 아래 항목이 분명해야 합니다.

- 이번 교재에서 반드시 남겨야 하는 산출물
- 단계별 마일스톤과 권장 순서
- 문서와 코드의 선후 관계
- 협업 시 지켜야 할 규칙
- 검증과 배포의 완료 기준

이 기준이 있어야 뒤 장의 실습이 단순 코딩이 아니라 재현 가능한 프로젝트 진행 방식이 됩니다.

# 1. 이번 교재의 필수 산출물
이 교재는 코드만 작성하는 강의가 아닙니다. 설계와 검증까지 남는 프로젝트를 만드는 강의입니다. 따라서 아래 산출물은 반드시 남겨야 합니다.

## 1.1 기획 산출물
- 서비스 비전과 문제 정의
- 사용자/액터/핵심 시나리오
- MVP 범위와 제외 범위
- 화면 목록과 사용자 여정 요약

이 산출물은 "왜 이 기능이 필요한가"를 설명하는 기준입니다.

## 1.2 도메인 설계 산출물
- 바운디드 컨텍스트 정의
- 애그리거트, 엔티티, 값 객체 정리
- 유스케이스별 도메인 규칙
- 컨텍스트 간 상호작용 메모

이 산출물은 뒤에서 API와 DB 구조를 설계할 때 기준이 됩니다.

## 1.3 화면 설계 산출물
- storefront 화면 목록
- admin 화면 목록
- 주요 상태별 화면 설명
- 핵심 사용자 플로우

이 산출물은 프론트 구현 순서를 정하는 기준입니다.

## 1.4 구현 산출물
- frontend 코드
- backend 코드
- infra/runtime 설정
- AI 상담 챗봇 관련 코드와 설정

이 산출물은 실제 실행 가능한 결과물입니다.

## 1.5 검증 산출물
- 기능 검증 체크리스트
- 실행 명령과 결과 기록
- 핵심 시나리오 검증 메모
- 배포 후 확인 기록

이 산출물이 있어야 "동작한다"는 말을 근거 있게 할 수 있습니다.

# 2. 산출물의 선후 관계
이번 교재에서는 아무 순서로나 만들지 않습니다. 산출물은 아래 순서를 따릅니다.

1. 기획 산출물
2. 도메인 설계 산출물
3. 화면 설계 산출물
4. 개발 환경과 인프라 설정
5. backend 골격 구현
6. frontend 골격 구현
7. 핵심 시나리오 단위의 기능 구현
8. 검증과 배포 기록

이 순서를 따르는 이유는 간단합니다.

- 기획 없이 도메인 설계를 할 수 없다
- 도메인 설계 없이 API와 화면 경계를 잘 잡기 어렵다
- 환경과 인프라가 없으면 구현과 검증이 재현되지 않는다
- 검증 기록이 없으면 배포 상태를 신뢰할 수 없다

즉 문서와 코드는 대립 관계가 아니라, 앞 산출물이 뒤 산출물의 입력값이 되는 관계입니다.

# 3. 이번 교재의 권장 마일스톤
이번 교재는 아래 마일스톤 순서로 진행하는 것이 가장 자연스럽습니다.

## 마일스톤 1. 서비스 기획 고정
- 비전 정리
- 사용자/액터 정리
- MVP 고정
- 산출물 구조 합의

### 완료 기준
- 무엇을 만들고 무엇을 만들지 않을지가 문서로 정리되어 있다

## 마일스톤 2. DDD 모델링
- 바운디드 컨텍스트 정의
- 애그리거트 경계 정의
- 도메인 규칙 정리

### 완료 기준
- Catalog, Cart, Order, Payment, Admin, Support Assistant 같은 핵심 경계가 설명 가능하다

## 마일스톤 3. 화면 기획
- storefront 화면 설계
- admin 화면 설계
- 챗봇 인터랙션 설계

### 완료 기준
- 핵심 시나리오별 진입 화면과 완료 화면을 설명할 수 있다

## 마일스톤 4. 개발 환경과 인프라 셋업
- 모노레포 구조 확정
- frontend/backend 프로젝트 초기화
- 로컬 실행 경로 구성
- 배포 가능한 기본 런타임 구성

### 완료 기준
- 팀원 또는 수강생이 같은 방식으로 프로젝트를 띄울 수 있다

## 마일스톤 5. 핵심 기능 구현
- 인증
- 카탈로그
- 장바구니
- 주문/결제
- 운영자 대시보드
- 챗봇 기본 흐름

### 완료 기준
- 핵심 시나리오가 실제 화면과 API를 통해 이어진다

## 마일스톤 6. 검증과 배포
- 핵심 시나리오 검증
- 빌드 확인
- 배포 반영
- 배포 후 재확인

### 완료 기준
- 로컬과 배포 환경에서 핵심 흐름이 재현된다

# 4. 일정은 어떻게 잡아야 하는가
이 교재는 실제 팀 프로젝트처럼 진행할 수 있도록 설계되어 있습니다. 정확한 날짜보다 중요한 것은 단계별 시간 배분입니다.

권장 비율은 아래와 같습니다.

| 단계 | 권장 비율 | 이유 |
| --- | --- | --- |
| 기획 | 15% | 뒤 수정 비용을 크게 줄여 준다 |
| 도메인 설계 | 15% | API와 코드 경계의 질을 결정한다 |
| 화면 설계 | 10% | 프론트 구현 속도와 일관성을 높인다 |
| 인프라/셋업 | 10% | 재현 가능한 개발 환경을 만든다 |
| 구현 | 35% | 실제 기능을 완성하는 중심 단계 |
| 검증/배포 | 15% | 완성도와 신뢰도를 결정한다 |

중요한 점은 구현 시간이 가장 길더라도, 기획과 설계를 너무 줄이면 전체 품질이 떨어진다는 것입니다.

# 5. 협업 규칙
이번 교재는 개인 학습에도 쓸 수 있지만, 팀 실습에도 바로 적용할 수 있도록 협업 규칙을 명시합니다.

## 5.1 문서가 먼저, 코드는 그다음
요구사항이나 경계가 바뀌면 먼저 관련 문서를 수정합니다. 문서와 코드가 충돌할 때는 "최근 코드"가 아니라 "합의된 기획과 설계"를 기준으로 정리해야 합니다.

## 5.2 기능은 시나리오 단위로 나눈다
작업 분할은 파일 기준보다 시나리오 기준이 좋습니다.

예를 들어:

- 상품 탐색 시나리오
- 로그인 시나리오
- 장바구니 시나리오
- 주문/결제 시나리오
- 운영자 주문 모니터링 시나리오

이렇게 나누면 프론트와 백엔드가 같은 목표를 보고 움직일 수 있습니다.

## 5.3 완료의 기준은 코드 작성이 아니다
아래 네 가지가 모두 있어야 완료라고 부를 수 있습니다.

- 관련 문서가 갱신되어 있다
- 코드가 구현되어 있다
- 실행 또는 테스트 결과가 있다
- 핵심 시나리오 기준으로 검증이 끝나 있다

## 5.4 위험한 변경은 경계를 먼저 다시 본다
인증, 주문, 결제, 외부 연동, 챗봇 응답 범위처럼 경계를 건드리는 변경은 바로 구현하지 말고 먼저 도메인 규칙과 시나리오를 다시 확인해야 합니다.

## 5.5 공통 규칙을 눈에 보이게 남긴다
- 브랜치 전략
- 커밋 규칙
- 환경 변수 규칙
- 테스트 실행 규칙
- 배포 확인 규칙

이런 항목은 암묵적으로 공유하지 말고 문서로 남기는 것이 좋습니다.

# 6. 검증 규칙
이번 교재는 "나중에 테스트" 방식으로 진행하지 않습니다. 시나리오 완성과 검증은 같이 갑니다.

## 6.1 무엇을 검증해야 하는가
- 상품 탐색이 되는가
- 인증이 되는가
- 장바구니 상태가 유지되는가
- 주문 생성과 결제 상태 변화가 보이는가
- 운영자 화면에서 결과를 확인할 수 있는가
- 챗봇이 정해진 범위 안에서 응답하는가

## 6.2 언제 검증해야 하는가
- 기능 하나를 붙인 직후
- 핵심 시나리오 한 묶음이 완성된 시점
- 배포 직후

## 6.3 무엇으로 완료를 판단하는가
- 빌드 성공만으로는 부족하다
- API 단위 호출만으로도 부족하다
- 실제 사용자 흐름 기준으로 끝까지 확인해야 한다

이 원칙이 중요합니다. 이커머스는 상태가 바뀌는 서비스이므로, 읽기 API만 확인해서는 완료를 말할 수 없습니다.

# 7. 배포 규칙
이 교재에서는 배포도 구현의 마지막 단계가 아니라 검증의 일부로 봅니다.

## 기본 원칙
- 로컬에서 먼저 검증한다
- 그다음 원격 환경에 반영한다
- 배포 후 핵심 시나리오를 다시 확인한다

## 배포 후 반드시 다시 보는 것
- storefront 기본 진입
- 로그인
- 장바구니
- 주문/결제 결과
- admin 진입
- 챗봇 기본 응답

즉 배포 성공 메시지가 완료가 아니라, 배포 후 사용자 흐름 재확인이 완료 기준입니다.

# 8. 실습용 운영 체크리스트
이 장을 따라 하는 학습자는 아래 체크리스트를 프로젝트 초반에 만들어 두는 것이 좋습니다.

- 기획 문서가 있다
- 도메인 설계 문서가 있다
- 화면 설계 문서가 있다
- frontend/backend/infra 초기화가 끝났다
- 핵심 시나리오별 구현 순서가 정리되어 있다
- 테스트와 실행 명령이 문서화되어 있다
- 배포 후 확인 항목이 정리되어 있다

# 9. 자주 하는 실수
- 기획 문서를 한 번 쓰고 다시 보지 않는다
- 설계 없이 바로 화면과 API를 동시에 만든다
- 구현이 끝난 뒤에야 검증 기준을 생각한다
- 배포를 끝으로 보고 배포 후 사용자 흐름을 확인하지 않는다
- 협업 규칙을 말로만 정하고 문서로 남기지 않는다

이 다섯 가지 실수는 후반부에 일정 지연과 품질 저하로 바로 이어집니다.

# 정리
이 페이지에서 우리는 이 프로젝트를 어떤 순서와 어떤 기준으로 굴릴지 정했습니다. 이제 이 강의는 단순한 기능 구현 목록이 아니라, 기획 -> 설계 -> 구현 -> 검증 -> 배포의 선후 관계가 분명한 실전 교재가 됩니다. 다음 장부터는 이 기준을 바탕으로 기획을 DDD 도메인 언어로 번역하기 시작합니다.
