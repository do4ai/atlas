# 3. DB, 스토리지, 시크릿, 실행 스크립트
앞 페이지에서 로컬 개발환경과 `compose`, `env` 구조를 정했다면, 이제 실제 데이터를 어디에 저장하고 어떤 값을 비밀로 다루며 어떤 명령을 반복 실행할지까지 정해야 합니다. 이 단계가 빠지면 프로젝트는 겉으로는 실행되는 것처럼 보여도 금방 무너집니다.

처음에는 메모리 저장소로 대충 돌아가고, 개발자 로컬 파일에만 값을 넣어도 되는 것처럼 느껴질 수 있습니다. 하지만 주문, 결제, 운영자 조회, 챗봇 상담 기록이 붙기 시작하면 바로 문제가 드러납니다.

- 서버를 재시작하면 주문 데이터가 사라진다
- 상품 이미지나 첨부 파일 경로가 사람마다 다르다
- 결제 시크릿이나 세션 키가 개인 PC에만 남아 있다
- 실행 명령이 제각각이라 새로 들어온 사람이 재현하지 못한다

이번 페이지의 목적은 이 혼선을 없애는 것입니다. 이커머스 서비스에 필요한 최소 저장 전략, 시크릿 주입 원칙, 실행 스크립트 구성을 먼저 고정하고 나서 다음 장의 실제 구현으로 들어갑니다.

# 왜 이 페이지가 구현 전에 필요할까
풀스택 강의에서 자주 생기는 착각은 "일단 API부터 만들고 나중에 DB 붙이면 된다"는 생각입니다. 물론 아주 짧은 데모라면 가능하지만, 이번 교재는 DDD 기반 이커머스 서비스를 끝까지 완성하는 흐름입니다. 따라서 아래 세 가지는 구현 전에 먼저 정해두는 편이 맞습니다.

- 어떤 데이터를 영속 저장할 것인가
- 어떤 값은 비밀로 다뤄야 하는가
- 어떤 반복 명령을 스크립트로 고정할 것인가

이 셋이 먼저 정리되어야 이후의 백엔드 유스케이스와 프론트 흐름도 안정적으로 이어집니다.

# 이 페이지에서 정할 것
이 페이지를 읽고 나면 아래 항목이 정리되어 있어야 합니다.

- 이번 교재에서 사용할 DB 전략
- 파일 업로드와 정적 자산의 저장 위치
- 시크릿과 설정값의 주입 방식
- 반복 실행 명령의 스크립트 구조
- 개발자와 운영자가 같은 기준으로 재현할 수 있는 실행 방식

# 1. 이번 교재에서 어떤 데이터를 영속 저장할까
이커머스 서비스에서는 모든 데이터를 똑같이 다룰 수 없습니다. 어떤 것은 메모리에 있어도 되지만, 어떤 것은 반드시 DB에 남아야 합니다.

이번 교재에서 최소한 영속 저장해야 하는 것은 아래와 같습니다.

- 사용자 계정
- 상품 정보
- 장바구니 상태
- 주문
- 결제 결과
- 운영자 조회를 위한 주문 요약 정보
- AI 상담과 관련된 최소 로그 또는 문의 맥락

반대로 아래는 초기 단계에서는 반드시 DB에 넣지 않아도 됩니다.

- 화면 장식용 정적 카피
- 완전히 하드코딩 가능한 데모 배너
- 변하지 않는 단순 상수 목록

중요한 점은 "서비스 상태를 복원하는 데 필요한 것"은 DB에 남겨야 한다는 것입니다.

# 2. DB는 무엇을 선택하는가
강의용 프로젝트라고 해서 무조건 가벼운 DB를 쓰는 것이 정답은 아닙니다. 반대로 실전 같아 보인다고 처음부터 너무 무거운 구성을 넣는 것도 좋지 않습니다.

이번 교재에서는 아래 원칙으로 판단하는 것이 좋습니다.

## 로컬 개발과 CI에서 재현하기 쉬워야 한다
- 누구나 같은 방식으로 띄울 수 있어야 한다
- 초기 데이터와 테스트 데이터 주입이 쉬워야 한다

## 주문과 결제 상태를 구조적으로 다룰 수 있어야 한다
- 트랜잭션 처리 개념을 설명할 수 있어야 한다
- 관계형 모델링이 자연스러워야 한다

## 나중에 확장 가능한 구조여야 한다
- MVP에서는 단순해도, 운영 관점의 사고를 막지 않아야 한다

이 기준을 적용하면 이번 교재는 관계형 DB를 기본으로 두는 것이 자연스럽습니다. 구현 난이도와 실습 안정성을 고려해 초기에는 `SQLite` 또는 `PostgreSQL` 중 하나를 선택할 수 있지만, "DDD 기반 이커머스"라는 목표를 생각하면 관계형 모델이 맞습니다.

# 3. 왜 관계형 DB가 자연스러운가
이번 프로젝트는 아래 관계를 자주 다룹니다.

- 한 사용자는 여러 주문을 가진다
- 한 주문은 여러 주문 항목을 가진다
- 한 장바구니는 여러 장바구니 항목을 가진다
- 한 주문은 하나 이상의 결제 시도와 연결될 수 있다

이 구조는 문서형 DB보다 관계형 DB에서 설명과 검증이 더 쉽습니다. 특히 아래 장점이 큽니다.

- 주문과 주문 항목의 관계를 명확히 표현하기 쉽다
- 상태 전이와 제약을 설명하기 쉽다
- 운영자 조회용 쿼리를 만들기 좋다
- 테스트 데이터 정리와 초기화가 단순하다

즉 이번 교재에서는 관계형 DB가 "실전형 사고를 연습하기 좋은 기본 선택"입니다.

# 4. 이번 교재에서 DB에 들어갈 핵심 테이블
정확한 스키마는 뒤 장에서 더 구체화되지만, 지금 시점에서 최소한 아래 테이블을 예상할 수 있어야 합니다.

- `users`
- `products`
- `cart_items`
- `orders`
- `order_items`
- `payments`
- `admin_users`
- `support_conversations` 또는 `support_events`

중요한 것은 지금 완벽한 컬럼을 다 정하는 것이 아니라, 어떤 도메인 정보가 결국 영속 저장 대상인지 인식하는 것입니다.

# 5. 파일 스토리지는 어디까지 필요한가
초기 이커머스 MVP에서도 파일 저장은 생각보다 빨리 필요해집니다. 대표적으로 아래 경우가 있습니다.

- 상품 이미지
- 운영자용 업로드 파일
- 챗봇이 참고할 FAQ 원본 파일

하지만 모든 파일 저장을 처음부터 복잡하게 만들 필요는 없습니다. 이번 교재의 초기 기준은 아래처럼 가져가면 충분합니다.

## 정적 자산
- 프론트 앱 리포지토리 안의 정적 에셋 또는 CDN 경로

## 운영 중 업로드되는 파일
- 로컬 개발에서는 지정 디렉터리 볼륨
- 배포 단계에서는 외부 오브젝트 스토리지로 확장 가능하도록 경계 유지

즉 처음에는 로컬 디렉터리 기반 저장으로 시작하되, 코드 구조는 "나중에 S3류 스토리지로 바꿀 수 있게" 분리해두는 것이 맞습니다.

# 6. 저장 경로는 왜 명시적으로 정해야 하는가
파일 저장에서 가장 흔한 문제는 경로가 암묵적이라는 점입니다.

예를 들어:

- 어떤 사람은 `uploads/`
- 어떤 사람은 `/tmp/uploads`
- 어떤 사람은 앱 루트 밖 개인 폴더

이렇게 되면 실행과 테스트가 깨집니다. 따라서 이번 교재에서는 파일 저장 경로도 설정으로 관리하는 것이 좋습니다.

예를 들면 아래와 같습니다.

- `STORAGE_ROOT`
- `PRODUCT_IMAGE_DIR`
- `CHATBOT_KB_DIR`

로컬에서는 하나의 명확한 루트 아래 두고, `compose`에서는 볼륨으로 연결하는 방식이 실용적입니다.

# 7. 시크릿은 무엇이고 왜 분리해야 하는가
시크릿은 단순히 "남에게 보여주면 안 되는 문자열"이 아닙니다. 유출되면 세션 위조, 결제 오용, 관리자 계정 침해, 외부 API 오남용이 가능한 값입니다.

이번 교재에서 시크릿 후보는 아래와 같습니다.

- `SECRET_KEY`
- 세션 서명 키
- JWT 서명 키를 쓴다면 해당 키
- 결제 연동용 비밀 키
- AI API 연동 키
- 관리자 초기 비밀번호 또는 초기화 토큰

이 값들은 코드에 하드코딩하면 안 됩니다. 문서에도 실제 값을 넣으면 안 됩니다. `.env.example`에는 이름과 형식만 있고, 실제 값은 각 환경에서 주입되어야 합니다.

# 8. 시크릿과 일반 설정값을 구분하는 기준
모든 env가 시크릿은 아닙니다. 이 구분이 안 되면 보안과 운영이 동시에 지저분해집니다.

## 시크릿으로 다뤄야 하는 것
- 외부에서 악용 가능한 인증/서명 값
- 외부 API private key
- 관리자 계정 초기화에 영향을 주는 값

## 일반 설정값으로 다뤄도 되는 것
- 앱 포트
- 호스트명
- 기능 플래그
- base URL
- 저장 경로

즉 `APP_PORT`와 `SECRET_KEY`는 같은 env에 있을 수는 있어도, 같은 민감도로 다뤄서는 안 됩니다.

# 9. 시크릿은 어디서 읽게 할까
시크릿은 아래 계층에 퍼지면 안 됩니다.

- 프론트 코드
- 공개 저장소 문서
- 테스트 스냅샷

이번 교재에서는 최소한 아래 원칙을 지키는 것이 맞습니다.

- 시크릿은 서버 런타임에서만 읽는다
- 프론트는 필요한 공개 설정값만 읽는다
- 서버는 시크릿을 설정 객체 한 곳에 모아 읽는다
- 로그에 시크릿 값을 출력하지 않는다

이 원칙만 지켜도 초반 보안 실수를 많이 줄일 수 있습니다.

# 10. 실행 스크립트는 왜 따로 둬야 하는가
강의형 프로젝트에서 반복 실행 명령이 문서에만 있으면 재현성이 떨어집니다. 사람마다 명령을 다르게 치기 시작하면 곧바로 "누구 환경에서는 되는데 누구 환경에서는 안 되는" 상태가 됩니다.

따라서 자주 반복하는 작업은 `scripts/` 아래에 고정하는 편이 좋습니다.

이번 교재에서 스크립트로 분리하기 좋은 작업은 아래와 같습니다.

- 개발 서버 시작
- 테스트 실행
- 린트 실행
- DB 마이그레이션 또는 초기화
- 시드 데이터 주입
- 샘플 관리자 계정 생성

문서에는 스크립트를 설명하고, 실제 실행은 스크립트를 통해 하게 만드는 것이 좋습니다.

# 11. 이번 교재에서 필요한 최소 실행 스크립트
아래 정도면 강의 진행과 실습 재현에 충분합니다.

## 루트 기준
- `scripts/dev/up.sh`
- `scripts/dev/down.sh`
- `scripts/test/all.sh`

## 서버 기준
- `scripts/server/seed.sh`
- `scripts/server/reset_db.sh`
- `scripts/server/create_admin.sh`

## 프론트 기준
- `scripts/client/storefront_dev.sh`
- `scripts/client/admin_dev.sh`

스크립트 이름은 팀 규칙에 따라 달라도 되지만, 역할은 명확해야 합니다.

# 12. 어떤 명령은 스크립트로 만들고 어떤 명령은 문서로만 둘까
모든 것을 스크립트로 만들 필요는 없습니다. 반복성과 실수 가능성으로 판단하면 됩니다.

## 스크립트로 만들어야 하는 것
- 여러 명령을 같은 순서로 실행해야 하는 작업
- 매번 옵션을 정확히 맞춰야 하는 작업
- 신규 참여자가 자주 실행할 작업

## 문서 설명만으로도 되는 것
- 한 번만 보는 개념 설명
- 의사결정 이유
- 구조 해설

즉 문서는 "왜"를 설명하고, 스크립트는 "어떻게 반복 실행할지"를 고정합니다.

# 13. DB 초기화와 시드 전략도 미리 정해야 한다
실전 교재에서 빠지기 쉬운 부분이 초기 데이터 전략입니다. 하지만 상품도 없고 관리자 계정도 없으면 프론트와 운영 화면을 제대로 확인할 수 없습니다.

따라서 이번 교재에서는 아래를 미리 정해두는 것이 좋습니다.

- 최소 상품 시드 데이터
- 테스트용 일반 사용자
- 테스트용 관리자 사용자
- 결제 mock 기본 상태

이 데이터는 사람이 수동으로 매번 넣기보다 스크립트로 재현 가능해야 합니다.

# 14. AI 상담 챗봇과 저장 전략은 어떻게 연결되는가
AI 상담 기능도 저장 전략과 무관하지 않습니다. 최소한 아래 질문에 답할 수 있어야 합니다.

- FAQ 원본은 어디에 저장되는가
- 대화 로그를 저장할 것인가
- 저장한다면 얼마 동안 보관할 것인가
- 주문 조회 맥락을 챗봇이 참조할 수 있는가

초기 MVP에서는 모든 대화를 완전하게 보존하지 않아도 되지만, 최소한 문의 맥락과 FAQ 소스는 추적 가능해야 합니다. 그래야 운영자가 챗봇 품질을 개선할 수 있습니다.

# 15. 이번 페이지를 구현 산출물로 바꾸면 무엇이 나와야 하나
이 페이지를 실제 작업으로 옮기면 아래 산출물이 있어야 합니다.

- DB 선택 문서 또는 README 섹션
- 기본 스키마 방향 표
- `.env.example`
- 로컬 업로드 디렉터리 규칙
- `scripts/` 하위 실행 스크립트
- 초기 데이터 시드 규칙

이 산출물이 있어야 다음 장에서 "이제 백엔드를 구현하자"는 말이 현실적으로 성립합니다.

# 16. 자주 하는 실수
- 주문과 결제 데이터를 메모리 저장으로 오래 끌고 간다
- 파일 저장 경로를 사람마다 다르게 쓴다
- `SECRET_KEY`를 코드에 하드코딩한다
- 실제 비밀값을 `.env.example`에 넣는다
- 반복 실행 명령을 문서에만 적고 스크립트로 고정하지 않는다
- 시드 데이터 없이 구현을 시작한다

이 여섯 가지 실수는 초반에는 티가 안 나지만, 장바구니와 주문 기능이 붙는 순간 바로 개발 속도를 떨어뜨립니다.
